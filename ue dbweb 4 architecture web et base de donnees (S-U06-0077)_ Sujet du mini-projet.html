Sujet du mini-projet

EXERCICE DE L'ETUDIANT (LOUIS RIVES-LEHTINEN) :

Projet :
login.php : Se connecter avec le prénom nom et le mdp est l'id du client. Se connecter nous dirige directement sur la page commande.php de l'utilisateur. Selon si l'utilisateur est admin ou non, il débloquera certains accès sur sa page de commande à travers des liens (ex: la page client.php ainsi que magasinsparproduit.php et etatdescommandes.php)
client.php

Mini-projet -- Site myBike, compléments

L'objectif du mini-projet est de compléter la version préliminaire d'un site de gestion de commandes de clients de magasins de cyclisme, débuté lors du TP 4. Les différentes améliorations proposées sont listées ci-dessous.

Attention : Le rendu du mini-projet devra être fait au plus tard la veille du TP noté 2 : une archive déposée dans "Rendu du mini-projet" et une version disponible en ligne sur pedago.univ-avignon.fr (le lien vers ce site doit être ajouté en commentaire au début du fichier clients.php).

Partie 1 - Modélisation des classes

Dans un premier temps un modèle de classe doit être implémenté pour les données.

A faire

  1. A partir des caractéristiques fournies par la base de données, établir le schéma de classe pour le projet.
    Attention : les classes PHP reflètent les objets ISSUS de la base de données pour le programme : i.e. les données telles qu'on les utilisent dans le PHP, pas forcément exactement comme elles sont dans la BD.
    Attention : et donc ces classes devront être conçues en anticipant les modifications du programme demandées dans la Partie 2 (création des nouvelles pages).
  2. Proposer une visualisation du modèle (qui sera déposé dans un fichier à part lors du rendu). Par exemple avec yuml.me, ou au pire une photo d'un graphe manuel.
  3. Modifier la méthode _toString de chaque classe, afin de régler l'affichage en HTML de chaque classe

Partie 2 - Intégration PHP/PDO

Il s'agit ici de compléter les fonctionnalités du site initial après avoir adapté le site aux nouvelles classes élaborées ci-dessus.

A faire

  1. Transformer la version du site du TP 4 pour prendre en compte la nouvelle modélisation en classes et récupérer les données de la BD sous forme d'objets (FETCH_CLASS).
  2. Créer une nouvelle page etatdescommandes.php pour afficher un tableau avec toutes les vélos possibles (products) classés par le nombre de ventes.
  3. Chaque produit possède un lien vers une nouvelle page magasinsparproduit.php qui affiche la liste des magasins qui ont acheté ce produit. Et pour chaque magasins le nombre de commandes de ce produit (tout statut de la commande compris).
  4. On ajoute ensuite des colonnes dans magasinsparproduit.php permettant : 1/ d'afficher le nombre de produits en commandes en cours (status) ; 2/ d'afficher le stock global correspondant (pour tous les magasins).
  5. Intégrer une gestion des droits d'accès : maintenant seuls les clients connectés au site peuvent voir l'état des commandes. Les identifiants sont le nom pour le login et l'id pour le mot de passe. De plus les clients de la base ont maintenant un statut optionnel 'admin' (binaire). Seuls les utilisateurs admin peuvent accéder à la visualisation de la page magasinsparproduit.php.  
    Attention : la prise en compte du statut conduit à modifier la BD. Penser a bien mettre la ligne SQL utilisée en commentaire dans votre fichier .php. Et a sauvegarder une nouvelle version de la BD (comme dans le fichier fourni au TP4, avec un dump en CLI) pour la recharger en cas de problèmes.
  6. Permettre l'ajout et la modification des clients : depuis la page clients.php, proposer l'accès à un modal dans lequel un administrateur (client avec un statut admin) peut saisir un id client. Si cet id n'existe pas : un nouveau client est créé et ses champs sont saisis. Si l'id existe, il s'agira d'une mise à jour (affichage des ses champs puis modifications possibles, changement de statut par exemple). Les comptes admin sont protégés, aucune mise à jour ne doit être possible sur ces clients.
    Attention : la logique voudrait que les comptes admin soient plutôt associés aux personnels des magasins (staffs). Nous avons choisis de rester sur les clients pour simplifier les requêtes DB nécessaires.  

Partie 3 - Amélioration UX jQuery

Dans cette dernière partie il s'agit de reprendre certaines pages du site pour améliorer l'expérience utilisateur en lui ajoutant des fonctionnalités d'affichage dynamique par le biais de la librairie jQuery.

A faire

  1. Dans la page commandes.php, le passage de la souris sur la ligne d'une commande provoque un passage en jaune de la ligne. 
  2. Dans la page clients.php, lors du passage de la souris sur un client avec un statut "admin" un zoom automatique est fait sur son nom pour en faciliter la lecture.

Consignes

Les fonctions du site seront developpées en HTML, CSS, JavaScript et PHP (à l'exclusion de tout autre langage).

La base de données est hébergée sur pedago.univ-avignon.fr.

L'aspect "physique" du site sera réalisé principalement avec W3.CSS. Le site devra être responsive : adapter son affichage en fonction de la taille de l'écran. Pour cela prévoir les 2 cas extremes (grand écran et smartphone) et montrer les modifications introduites en passant d'un cas à l'autre.

Une fois les fonctions précédentes réalisées, tout ajout pourra être envisagé dans la mesure où son apport au site est justifié.
